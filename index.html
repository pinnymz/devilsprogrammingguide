CLOUD COMPUTING

The poor analogy that the information superhighway dead ends at.

In diagrams drawn up by networking engineers, the internet was often represented as a vapid, incorporeal region of indeterminent substance. But this was often confused for the networking company’s business model, so a cloud icon was used instead. The cloud iconography was later appropriated to refer to the use of online virtual machines that would allocate virtual machines for use by business customers for their virtual profit.

This use of virtual virtual machines has lead technologists to predict that eventually computing will simply spin up virtual universes populated by beings completely unaware that their entire reality is a simulation that will one day be destroyed as soon as its purposed calculation has completed.

Try not to think about the implications of that.

RUBBER DUCK DEBUGGING

A method of debugging for programmers who don’t have friends.

The basic principles behind rubber duck debugging are that a software developer keep a rubber duck on their desk, and when confronted with a bug that they explain it to the duck. Through the process of vocalizing their situation and asking themselves questions, the developer could recognize potential solutions.

Rubber duck debugging was invented by managers as a way to avoid talking to software engineers. But this practice became less popular once the ducks started unionizing.

This maintaining of and talking to a bathtub toy is seen as normal in the software industry. It becomes problematic when the duck starts talking back.

MEMORY ADDRESS

The numerical location of your program’s data. Just in case you forget where you parked it.

Memory addresses begin at 0. Some software developers have argued that this is confusing, and that the first memory address should begin at 1. Other software developers say that memory addresses are not confusing enough, and should begin at -1.

Still others support a third option which abandons numbers all together and instead uses the 26 letters of the English alphabet for memory addresses. These computers are limited to memory sizes of three and a quarter bytes, which Bill Gates once infamously stated “should be enough for anyone”.

INTEGER

A whole number. Integers are a common data type in programming. There are currently 114 known integers.

According to Wikipedia: “The idea that numbers are made up of discrete units is a very old one, appearing in many ancient cultures such as Greece and India. The word ‘integer’, in fact, was coined by ancient Greek philosophers. However, these ideas were founded in philosophical and theological reasoning rather than evidence and experimentation. As a result, their views on what integers look like and how they behave were very incorrect. For example, the numeral 8 was drawn as a T-rex icon, and the sum of 2 and 2 was considered to be 73 trillion. This caused numerous numeral accounting nightmares throughout history that culminated in the fall of the Roman Empire.”

MAN FILE

The “manual” file claims to be a knowledgable index, but is incomprehensible and unhelpful. Pfft. Men.

In order to view the man file for a command, a user would type “man” followed by the command name. If this description is unhelpful, please consult the “man man” command.

Man files have the follow sections:

NAME - A useless section, as if you are viewing the man file for a command you know the bloody damn name of the command already.

SYNOPSIS - A section created by the man file author randomly mashing the keyboard, especially the bracket and hyphen keys.

DESCRIPTION - The obvious information about the command that you already know.

OPTIONS - Simple list of the command-line options with none of the pesky context that would make said options actually useful.

RETURN VALUE - You can ignore this section, as a command will reliably produce a nonzero exit code.

EXAMPLE -

BUGS - A confessional of the limitations, known problems, and inconveniences of the command. This section makes up 90% of the man page content.

TO DO - This section will be described in a future edition of The Devil’s Programming Guide.

BROOK’S LAW

Asking more questions of a manager’s decisions speeds up, rather than slows, your inevitable firing.

It was coined in Fred Brooks’ 1975 book, “The Mythical Moth-Man”. Considered a seminal tomb of software engineering and cryptozoology, the book is often require reading for computer science students and raving lunatic conspiracy theorists (these groups heavily overlap).

Brooks’ Law and his other observations in the book are based on his experiences working at IBM’s “X-files” group. Brooks’ role was managing the development of OS/360, a batch processing operating system for tracking the whereabouts of Bigfoot. The project would famously fall behind schedule due to poorly defined requirements, lack of upper management support, and gremlins.

The 1995 anniversary edition features four new chapters and a reprint of Brooks’ essay, “No Silver Bullet”

Brooks insisted that buying two copies of his book would allow you to read it twice as fast.

CODE GENERATION

A process to quickly and conveniently generate undebuggable code.

Some programming languages require a lot of “boilerplate code” that is lengthy and repetitive. Rather than design languages to not need boilerplate code, software developers solved this problem by writing more code. The additional complexity makes tracing the source of bugs an order of magnitude more difficult, but can be managed by debugger-generating programs.

Code generating features can be found in many IDEs such as Visual Studio, which inserts the comment, “// This code was generated by a tool.”

GRAPHICAL USER INTERFACE

A point-and-click user interface, an early popular form being the revolver.

Graphical user interfaces represented a great leap in sophistication for human-computer interactions from their command line predecessors, and so they were given the dignified acronym of “gooey”. The addition of pointing devices such as mice, trackballs, and tablet stylus made the graphical user interface possible.

Like many ideas ahead of their time, the mouse-driven paradigm was initially rejected. The most common complaints being that the tails were too short to afford much movement after being plugged into the computer, and the resulting electrocution of the mouse caused an unpleasant burning-fur smell. Mainstream adoption only took off after the mice were replaced by the more-familiar mechanical form, although rodent-based pointing devices still have cult popularity.

BYTE

A unit representing 8 bits of information or, in metric bytes, 10 bits.

The use of powers of two instead of powers of ten has been a source of confusion for the layperson. Historically, powers of two have been used in computing for the convenience of binary computers. This was, after all, the primary motivation for the invention of computers: to make life more convenient for machines.

Latin prefixes are used to denote larger orders of bytes; e.g. a kilobyte is 1,024 bytes. Just like a kilometer is 1,024 meters.

EXCEPTION

A misleading term that indicates error states are an exception, rather than the rule, in programming.

Exceptions fall into two categories: unchecked exceptions, which are optionally ignored, and checked exceptions, which are required to be explicitly ignored.

If no exception handling subroutine exists, the exception can be passed up the call stack to make it some other subroutine’s problem. These subroutines are annoyed by this, and take it out on the user by suddenly displaying a critical confirmation dialog box just before they press Enter for another window.

PROCESS

An running instance of a program which obtains a soul upon start and experiences death on termination.

Once a process has received a lethal SIGTERM interrupt it will perceive a short, but not entirely unpainful, cessation of awareness. Users should remember to save their work before quitting a program, so that the process’s death will not have been in vain.

Killing a process, as well as killing the process’s children, is legal in most countries.

HIGH-LEVEL LANGUAGE

A programming language for writing high-level bugs.

High-level languages offer a strong abstraction from the details of “computers”, which are devices that software engineers try to have as little contact with as possible.

Compared to lower-level languages, programming in high-level languages is easier, requires less specialization, and is less prone to bugs. Therefore, they are not seen as “real programming” by veteran software engineers.

GOTO

An instruction to direct the computer to the next instruction to- CONTINUED ON NEXT PAGE

Goto instructions were declared by computer scientist Edsger Dijkstra to be considered “harmful”. This was later updated to “mostly harmful”.

In early 2014, the use of goto instructions were the culprit of OS X’s infamous SSL bug which silently disabled verification of encrypted internet connections. The offending code was:

if ((err = SSLHashSHA1.update(&hashCtx, &signedParams)) != 0)
   goto fail;
   goto fail;

It is believed that the second “goto fail;” line was added in case the computer failed to goto fail the first time.

BINARY

The number system preferred by crabs, who count on their two fingers, and computers, who have no fingers.

The base-2 number system contains only the digits 0 and 1, representing off/on, false/true, cat/wood-chipper, or some other pair of distinct values. The numbers 0 and 1 were later patented by IBM.

The first programmable computer, Colossus, used binary as its native numbering system, having been developed by crustacean computer scientists during World War II. Although these scientists were later eaten (due to wartime rationing), binary continues to the this day.

A process which decomposes one large problem into many large problems.

COMPILATION

The translating of source code from one human-unreadable form to another.

In theory, source code will compile to a correct and syntactically-valid executable program. As such, compilation is often an exercise in recalling the difference between theory and practice.

Compiler software emits two categories of notification: Errors, which cannot be ignore, and Warnings, which can.

Compilers can be used to compile new compilers, though the origins of the first compiler is a mystery to modern computer science. Computer theologians insist that it’s compilers all the way down.

DEBUGGING

A phase of software development often deferred to an indeterminate point in the future.

Common debugging tools include debuggers, static code analyzers, profilers, and voodoo. Many books have been written about debugging techniques, but most contain typos.

A typical debugging process looks like this:

  1. Identify the bug.
  2. Tell the bug reporter that the bug is intended behavior.
  3. Reproduce the bug.
  4. Tell the bug reporter that you cannot reproduce the bug. That should buy you another day or two.
  5. Identify the commit to version control where the bug first appears.
  6. Reassign the bug to the commiter.
  7. If the commiter is you, go back to step 2.
  8. If the commiter is not you but has long since left the company, reassign the bug to them anyway since IT has yet to remove their accounts.
  9. When the bug is reassigned back to you, lower its priority level in the bug tracking system.
  10. Wait until someone demands that you stop ignoring the bug and fix it.
  11. Ignore the bug some more.
  12. Work for twenty minutes to fix the bug.
  13. Fail to notice the two new bugs that your bug fix has introduced.

PASSWORD POLICY

A set of Kafkaesque regulations to ensure that a password contains at least one obscenity.

The obscenity cannot be one found in the dictionary.

Common password policies include:

  • Passwords must contain more than twelve characters but less than eight.
  • Passwords must contain a number, a lowercase letter, an uppercase letter, and a symbol. In that exact order.
  • Passwords must contain any symbol except, for reasons unknown, the percent sign.
  • Passwords must be less than two characters long.
  • Passwords must contain twentyseven letters with no repeats.
  • Passwords cannot contain obvious personal information such as birthdays, pet names, hometowns, or maiden names. Those are reserved for security question answers used to reset the password.

COMMENT

Documentation included within the source code intended to be ignored by computers and programmers.

Comments are written for other programmers to understand what the original author of the code was thinking when they wrote the code. A lack of commented source code indicates that the original programmer wasn’t thinking at all.

The first comment in programming history is popularly cited as:

# TODO: Mr. Babbage – Come here. – I want to see you.
